# 🟩 一、AI 使用与工作流

---

## **1. 你在开发中是如何使用 AI 工具的？**

### **标准结构**

* 先说 AI 在你工作中的定位（效率工具，不是替代决策）
* 列举 3–4 个典型使用场景
* 强调 review、测试、安全
* 最后强调人机协作的 workflow

### **示例回答**

> 我把 AI 作为提升效率的工具，不作为业务决策者。
> 在实际开发中主要用于：
> 1）生成模板代码和脚手架；
> 2）写正则、测试用例、类型定义；
> 3）辅助查文档、理解错误日志；
> 4）自动生成接口文档和注释。
>
> 但所有 AI 生成的内容我都会经过 **手动 review + 单测验证 + 性能检查**，确保逻辑、安全性、可维护性都符合要求。

---

## **2. AI 生成代码可能带来哪些风险？你如何避免？**

### **标准结构**

* 三类风险：逻辑、性能、安全
* 每类给一个真实场景
* 给出你如何避免的流程

### **示例回答**

> AI 生成代码主要有三类风险：
> **逻辑风险**：常出现业务约束缺失、边界 case 漏掉。
> **性能风险**：例如不必要的深拷贝、频繁 setState、O(n²) 循环。
> **安全风险**：如 XSS / 注入风险、未校验输入。
>
> 我的应对方法是：
> **人工审查 + ESLint + 单元测试 + 性能分析 + 小流量验证**。
> 所以我不会盲信 AI，而是把它当成“初稿生成器”。

---

## **3. 你如何评估 AI 生成代码的可信度？**

### **标准结构**

* 核心：不直接相信
* 三类验证：文档对比、测试验证、业务校验

### **示例回答**

> 我不会直接使用 AI 生成的代码，会从三个角度验证：
> 1）**文档对照**：是否符合框架官方最佳实践；
> 2）**测试验证**：写单测验证行为；
> 3）**业务验证**：检查是否符合实际业务边界条件。
>
> 只有完全符合工程规范后才会合入。

---

## **4. AI 会不会取代前端？你怎么看？**

### **标准结构**

* 明确“不取代，增强”
* 解释 AI 的局限
* 前端未来的发展方向

### **示例回答**

> AI 不会取代前端工程师，但会淘汰不会使用 AI 的工程师。
> AI 无法替代复杂业务理解、架构设计、安全/性能/可维护性判断，更无法承担产品责任。
>
> 前端未来会更多偏向：
>
> * AI 编排
> * 数据驱动的 UI
> * 工程化能力
> * 人机协作
>
> 所以 AI 是增强，不是替代。

# 🟩 二、AI 功能开发场景题

---

## **5. 如何在前端设计一个 AI Chat（流式输出）？**

### **标准结构**

* 接口 → 流式 → UI 渲染 → 状态管理 → 错误处理

### **示例回答**

> 我会使用 **fetch + ReadableStream** 从后端读取 AI 的流式输出，
> 前端使用逐段 append 的方式渲染，并控制：
>
> * 消息流式追加
> * AbortController 控制中断
> * 错误状态处理
> * markdown 渲染
>
> UI 层做虚拟滚动 + 打字机效果，保证性能和体验。

---

## **6. 前端如何处理大模型流式响应？**

### **示例回答**

> 采用 `response.body.getReader()` 读取分片，
> 每次读取后更新缓存和 UI，
> 避免 setState 过多导致卡顿，
> 可以用 ref 或 batch update。
>
> 在响应结束时关闭 reader，并处理错误和中断。

---

## **7. 如何实现 AI 自动补全（如搜索建议）？**

### **示例回答**

> 前端会做 debounce，避免频繁请求。
> 生成 prompt 时加入上下文（历史输入 / 当前页面语境）。
> 对每个 prompt 和结果做缓存（基于 hash）。
>
> UI 层处理：可编辑区域、灰色提示、键盘导航。
>
> 安全方面：输入合法性校验、防止 prompt 注入。

---

## **8. 如何实现 AI 智能表单生成？**

### **示例回答**

> 让 AI 输出 **JSON Schema**，
> 前端用一个表单渲染器动态渲染字段、验证规则、布局。
>
> 我还会加入：
>
> * 字段依赖
> * 自定义校验
> * 权限控制
> * Schema 可视化编辑器
>
> 输出必须做 schema 校验，避免 AI 生成非法结构。

---

## **9. AI 请求的 API Key 放前端安全吗？如果不能，该怎么做？**

### **示例回答（必考）**

> 绝对不能放前端，因为会被直接获取。
>
> 正确方案：
>
> * 前端 → 自家后端（无 key）
> * 后端代请求大模型 API，并加入限流、鉴权、日志
>
> 这是强制安全规范。

---

## **10. 如何防止 Prompt Injection？**

### **示例回答**

> 我会从三个层面防御：
> **前端**：输入校验、黑名单词、限制用户控制模板内容。
> **后端**：系统 Prompt 固化，用户 prompt 放到隔离区。
> **审核**：对输出做内容安全检查。
>
> 前后端双层隔离是关键。

# 🟩 三、AI + 前端框架（React/Vue）

---

## **11. AI 自动生成的 React 代码有哪些常见问题？**

### **示例回答**

> 常见问题包括：
>
> * 滥用 useEffect
> * 状态管理混乱
> * 缺少 key，导致 diff 异常
> * 性能差，如大循环放 UI 中
> * 未处理边界 case
>
> 都需要人工重构。

---

## **12. 你会如何审查 AI 生成的 React 组件？**

### **示例回答**

> 我的 checklist 是：
>
> * 状态是否可控
> * props 是否符合单向数据流
> * useEffect 是否产生副作用
> * 是否需要 memo / useCallback
> * 错误处理是否完整
> * 类型和边界条件是否覆盖
>
> 通过 checklist 审查安全性和可维护性。

# 🟩 四、性能优化 + AI

---

## **13. AI 生成的代码常见性能问题是什么？你怎么优化？**

### **示例回答**

> AI 常生成性能不佳的代码，例如：
>
> * map + filter + reduce 嵌套循环
> * 频繁 setState
> * 大对象 JSON 深拷贝
>
> 我的优化方向包括：
>
> * memoization
> * 虚拟列表
> * 分片渲染
> * Web Worker
> * 减少 re-render

---

## **14. AI 生成代码如何确保可维护性？**

### **示例回答**

> 通过以下规范化措施：
>
> * 拆分 UI / 逻辑 / hooks / service
> * 统一 lint + prettier
> * TS 强类型约束
> * 单元测试
> * 文档注释
>
> 让 AI 生成的代码符合团队工程规范。

# 🟩 五、AI + 系统设计

---

## **15. AI 返回大文本如何避免前端卡顿？**

### **示例回答**

> 我会使用 streaming 分段处理，
> 渲染层使用虚拟渲染 / 分块渲染，
> 复杂 markdown 转换放在 Web Worker，
> 必要时使用 incremental rendering。
>
> 多层优化保证不卡顿。

---

## **16. AI 的响应如何做前端缓存？**

### **示例回答**

> 我会对 prompt 做 hash，
> 缓存结构为：
> `{promptHash → AIResponse}`
> 存到 IndexedDB 或 memory cache。
>
> 还会设置 TTL 更新策略和前端 + 后端双层缓存。

---

## **17. 如何设计一个基于 RAG 的文档问答系统？**

### **示例回答（高分）**

> 前端负责：上传文件、渲染聊天、展示引用片段、缓存历史记录。
> 后端流程：
> 1）文件切片 → embedding
> 2）向量库检索
> 3）RAG 拼接 → 大模型生成
> 4）流式返回前端
>
> 关键是安全、权限、文件脱敏。

---

## **18. 大模型接口容易超时，你怎么处理？**

### **示例回答**

> * AbortController 中断
> * 超时自动重试 + 指数退避
> * loading + fallback
> * streaming 降低感知延迟
>
> UX 和稳定性同时兼顾。
